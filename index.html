BEM vs Atomic CSS

The dichotomy between
"best practice" and efficiency

Current best practices

  • Location independent: Do not nest unnecessarily
  • Aim for the lowest specificity for ease of future overwrites

Mixins in SASS

              
@mixin border-radius($radius) {
  -webkit-border-radius: $radius;
     -moz-border-radius: $radius;
      -ms-border-radius: $radius;
          border-radius: $radius;
}

.box { @include border-radius(3px); }
.totally-different-box { @include border-radius(10px); }

Mixins in your precompiler means less repetition in development, but they become so much more powerful in raw CSS.

Remember the BEM vs Atomic problem?

  • BEM is inelegant, clunky and repetitive
  • Atomic means holding styling information in semantic markup - bad for separation of duties

But what happens when we put it in raw CSS?

              
@mixin button($color) {
  padding: 1em;
  margin: 0 1em;
  background-color: $color;
}

@mixin display-font {
  font-family: 'Playfair Display';
  letterspacing: -0.01em;
  line-height: 1em;
}

.block--element {
  @include button(blue);
  @include display-font;
}

No styling in markup? Check.

Ease of prototyping? Check.

Explicity separating Settings, Layout and Decoration

              
.block--element_layout {
  all: initial;
  font-size: 19px;
  @include container(960px);
  border-color: blue;
}

.block--element_decoration {
  all: initial;
  @include display-font;
  color: pink;
  background: url('cat.jpg');
}

Separation of concerns

  • Separate properties that propagate down
  • Visually clear
  • Makes theming easier
  • A note on performance: reflows are much more expensive than repaints. This could make performance debugging clearer too.
  • Async loading

Scope and Inheritance

  • Define new scope with :scope
  • Protect your scope with all:inital
  • Clarify inheritance issues by separating out settings, layout and decoration
  • Scoping magic wands away specificity wars

Wrap up
