Our ideal language should eliminate specificity wars.

  • If we're talking about an optimal styling language, we would do very well to eliminate specificity wars, once and for all.
  • It doesn't matter how much good practice you know — when you're in a specificity war, good practice goes out the window

Wait, what is the cascade again?

The cascade is how the browser decides which CSS rules take precedent, depending on origin.

  • First of all, who here actually knows what the cascade is?
  • Bit of an imflammatory question to ask at CSSConf, I know
  • for the longest time I thought the cascade was inheritance
  • Wasn't until I was researching this talk did I learn the actual cascade order

The cascade:

  1. Select element; filter rules
  2. Cascade order W3C, MDN
    1. User agent
    2. User agent !important
    3. User
    4. Author
    5. CSS Animations (doesn't cascade)
    6. Author !important
    7. User !important
  3. Specificity
  • When the browser is rendering out the HTML and CSS, it's going through every element and creating nodes in the DOM.
  • For every element,

People are afraid of the cascade.

  • This confusion causes the classic !important declarations everywhere.
  • So I had long, deep think about how the cascade could be better, and y'know what?

The cascade is actually super useful.

  • We need a way to resolve conflicting rules

The ideal selector

  • What would our ideal selector look like?

Ideal selector criteria

  • Separation of concerns
  • Reusability (and DRY code)
  • Efficiency
  • What do we want?
  • Separation of concerns
  • There should be no DOM structure replicated in the CSS, nor should we see styling information encoded in the markup
  • Reusability
  • For ease of development, and also performance
  • Efficiency
  • I want to target only the elements I want, as fast and as unambiguously as I can.

    BEM = block element modifier

    Namespacing convention where your components define classes

    Declarations inside that component are not reused in other components

  • As an example, let's look at BEM and Atomic CSS
  • BEM is essentially a namespacing convention where your components define your classes
  • Declarations inside that component aren't shared with other components
  • Keeps code encapsulated, nice and tight

BEM vs Atomic CSS

    
.i { font-style: italic; }
.b { font-weight: bold; }
.underline { text-decoration: underline; }
.strike { text-decoration: line-through; }
.ttc { text-transform: capitalize; }
.ttu { text-transform: uppercase; }

    Atomic CSS = small reusable classes

    Classes define common rulesets

    Are applied to whichever element, whenever

  • Atomic CSS sits at the complete opposite of the spectrum
  • The philosophy being: classes should be general purpose; reusable.
  • Classes define common rulesets that can be used over and over again

Atomic CSS

  • Great for lightning past prototyping
  • Fails to separate content and style (styling in markup)
  • Lots of repetition as you apply 10, 20 classes to each element

BEM

  • No styling in markup, no repeated class calls
  • Repeats heavily in CSS as code is not reused across components
  • Can encode DOM structure in CSS class names
  • Now Atomic CSS is fantastic for lightning past prototyping, and if you haven't tried it then I highly reccommend giving it a chance
  • I know it seems weird, but trust me, you'll see its value when you making something with it
  • Problem is, all these classes in the markup means we're violating separation of concerns.
  • also lots of repetition in markup. Very DRY in CSS, though.
  • On the other hand, BEM doesn't leave styling in the markup, or repeating classes
  • But it does repeat very heavily in CSS since every ruleset is specific to a component
  • It's also common to see DOM structures popping up in CSS when using BEM

Reusability or Separation of Concerns; choose one

  • So it seems as if we've reached an impasse.
  • Can't have separation of content and style, while remaining DRY.

Functions in native CSS is incredibly powerful

              
                @function button($color) {
                  padding: 1em;
                  margin: 0 1em;
                  background-color: $color;
                }

                @function display-font {
                  font-family: 'Playfair Display';
                  letterspacing: -0.01em;
                  line-height: 1em;
                }

                .block--element {
                  function: button(blue);
                  function: display-font;
                }
  • Here, I'm defining two commonly used rulesets in my code as functions, and calling it in this class.
  • Notice here how there is no styling information in the markup, no DOM structure in CSS
  • I can reuse common bits of code as much as I want
